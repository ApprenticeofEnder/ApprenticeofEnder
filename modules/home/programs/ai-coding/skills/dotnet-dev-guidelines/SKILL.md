# .NET Development Guidelines

Follow these guidelines when planning or writing C# / .NET code. For review of an existing diff, see [`dotnet-code-quality`](../dotnet-code-quality/SKILL.md).

<rules>

## Code structure & simplicity

- Keep methods small: around 50 lines or less.
- Methods should have a single clear responsibility.
- Avoid deep nesting: prefer early returns and guard clauses.
- Limit class size: each class should have a single responsibility and be cohesive.
- Flatten complexity: use strategy pattern or polymorphism over giant switch/if trees.
- Partials should **only** be used to incorporate your code into an autogenerated class. Do not use partials to split up large classes or for any other reason.
- Use expression-bodied members for simple methods.
- Refactor as you go: when planning changes in a class or module, the first task should be to address some of the technical debt before building on top of it. Ideally methods being refactored are covered by tests to prove the refactor does not introduce breaking changes.

## Readability & maintainability

- No magic values, strings or numbers, ever — use constants and enums.
- Extend `enum` to realize custom string values, rather than calling a separate utility method, and also to return the `enum` corresponding to an input string.
- Descriptive naming: names should communicate intent without needing extra explanation.
  - Avoid abbreviations unless they are domain-standard.
  - Methods should be verbs; classes should be nouns.
- Self-documenting code: avoid unnecessary comments by writing clear logic. Document behavior using its corresponding unit test.
- Consistent naming conventions and code style across the codebase. Follow .NET conventions (PascalCase for types and methods, camelCase for parameters and local variables).
- Never use code regions — they hide code and encourage large, unwieldy classes and methods.
- Do not use underscores in names — this reduces readability.

## SOLID & dependency practices

- Dependencies and coupling are one of the biggest obstacles to maintainable and testable code. Following the Dependency Inversion Principle solves this.
- Program to interfaces / abstractions, not implementations.
- Follow the 5 SOLID principles:
  - **Single Responsibility:** a class should only have one responsibility or job.
  - **Open/Closed:** open for extension, closed for modification. Add new functionality without changing existing code, often through interfaces and inheritance.
  - **Liskov Substitution:** subclasses should stand in for superclasses without affecting correctness.
  - **Interface Segregation:** clients should not be forced to depend on members they don't use.
  - **Dependency Inversion:** high-level modules should not depend on low-level modules; both depend on abstractions. Don't `new` concrete types inside business logic; inject your dependencies. Don't depend directly on a type you can't mock — inject the dependency via a provider.

## Control flow & logic

- Avoid `else` when possible: prefer guard clauses and an early exit.
- Avoid long parameter lists — group into objects or records.
- No duplicate code — extract shared logic into a single method or utility class.
- Clear error handling: throw exceptions or return result types for predictable outcomes. Never fail silently.
- Don't return `null` for a collection. If you return null, the caller must remember to check for it or risk a `NullReferenceException`.

## Testability & robustness

- Code should be unit-testable. Dependencies should be explicitly injected, not created inside the class.
- Minimize or eliminate side effects. Prefer pure functions: same input → same output, no external side effects. Side effects (DB calls, file writes) should be isolated.
- Use mocks / fakes via DI. External dependencies (DB, APIs, file system) should be abstracted. In unit tests, replace them with mocks or fakes. Integration tests should use real dependencies to verify end-to-end functionality.
- Fail fast. Do not allow undefined or corrupted state to continue silently. Use exceptions or result objects to fail early with clear error messages.
- Immutable DTOs and models where appropriate.

## Lean on aspects of functional programming

- Favor expressions (which return a value) over long imperative statements.
- Prefer immutable data to reduce side effects and race conditions: `record` types or `init`-only setters.
- Use pure functions (output depends only on the arguments) where possible.

## Performance & scalability

- Use the **Benchmark** NuGet package, not `Stopwatch`, for performance analysis.
- Use `ValueTask` for high-performance scenarios.
- Avoid premature optimization:
  - Focus first on readable, maintainable code.
  - Profile actual bottlenecks before optimizing.
  - Optimize hot paths where performance matters most.
- Resource cleanup: properly release file handles, network streams, memory buffers, and disposable objects. Use `using` / `await using`.
- Use `Lazy<T>` to delay expensive initialization.
- Prefer `IEnumerable<T>` / `IAsyncEnumerable<T>` for deferred enumeration instead of materializing collections early.

## Asynchronous tasks

- Don't await independent tasks sequentially — run them concurrently with `Task.WhenAll()`.
- Always accept a `CancellationToken` in `async` methods, especially in services, repositories, and long-running operations.
- Use `IAsyncDisposable` for asynchronous resource cleanup.

## Logging

- Use structured logging instead of plain text logging.

## Testing framework

- Use **xUnit** for C# tests in this setup, unless the project specifies otherwise.

</rules>
